### SI-03.3 — VideosModule: entidade Video + public_id único
- **Status:** completed
- **Tests:** 10 passing
- **Observations:**
  - Used `nanoid@3` (CJS) instead of the pinned `^5.x` from library-refs.md. nanoid v5 is pure ESM; a dynamic `await import('nanoid')` failed in Jest with `TypeError: A dynamic import callback was invoked without --experimental-vm-modules`, since this project's Jest harness doesn't enable ESM support. library-refs.md itself flagged this exact fork ("use nanoid@3.x or dynamic import — flag this choice at implement time"); v3 has the identical `customAlphabet`/`urlAlphabet` API, so no code-shape change beyond the import.
  - Gave the entity's unique constraints explicit names (`UQ_video_public_id`, `UQ_video_storage_key`) via `@Unique(...)` instead of relying on TypeORM's autogenerated hash names, for readability — detection of which constraint fired still follows the existing codebase convention (`ChannelsService`'s `isPgUniqueViolationOnColumn`, matching on the Postgres error `detail` string), not the constraint name.
  - Found the `db` container fully detached from the Compose network (likely a Docker Desktop/WSL hiccup from a long-running environment) — `nestjs-api` and other services couldn't resolve `db` at all. Reattached it via `docker network connect --alias db` (no recreation, so the DB's existing data was preserved rather than lost, though in this case the DB turned out to have no schema yet).
  - Discovered the `db` container had no applied migrations at all (empty schema, no named volume) — ran `npm run migration:run` to apply the pre-existing Phase 1/2 migrations before generating this SI's migration, so the generated diff would be scoped to `videos` only.
  - Noted (not fixed, out of scope): a `pg` deprecation warning ("Calling client.query() when the client is already executing a query is deprecated") surfaces during the integration suites — pre-existing driver/TypeORM interaction, unrelated to this SI's code.
